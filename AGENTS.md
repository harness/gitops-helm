# AGENTS.md — gitops-helm

Helm chart for deploying the Harness GitOps Agent into customer Kubernetes clusters. Public chart published to https://harness.github.io/gitops-helm/.

## Project Overview

- **Type**: Helm chart repository (no application code)
- **Top-level chart**: `charts/` — `name: gitops-helm`, current version `1.2.6` (see `charts/Chart.yaml`)
- **Sub-chart dependency**: `argo-cd` from https://argoproj.github.io/argo-helm (vendored at `charts/charts/argo-cd-9.0.0.tgz`, see `charts/Chart.lock`)
- **Templates**: `charts/templates/{argocd-application-controller,argocd-applicationset,argocd-repo-server,gitops-agent,redis,upgrader,_common.tpl}`
- **Values**: `charts/values.yaml` is the primary user-facing API surface
- **Releases**: tarballs in `releases/` plus `index.yaml` are auto-generated by GitHub Actions on push to `main` (see `.github/workflows/release.yaml`)

## Build / Test / Lint

There is no compiled build — Helm templates are validated with `helm` directly.

```bash
# Refresh sub-chart deps (run after editing Chart.yaml dependencies)
helm dependency update charts/

# Static lint
helm lint charts/

# Render templates locally (with default values)
helm template gitops-helm charts/

# Render against a real values override
helm template gitops-helm charts/ -f my-values.yaml

# Test install into a cluster (--debug + --dry-run is the safest pre-flight)
helm install gitops-helm charts/ -n argocd --create-namespace --dry-run --debug
```

There is no unit-test framework wired up (e.g. no `helm-unittest`). Validation is via `helm lint` + manual rendering + deploying to a devspace cluster.

## Release Flow

- `charts/Chart.yaml` `version:` is the source of truth for chart version.
- Bump `version:` in the same PR as user-facing changes; CI on push to `main` packages and publishes via `helm/chart-releaser-action@v1.6.0`.
- Released tarballs live under `releases/` and are referenced from `index.yaml`.
- Update `CHANGELOG.md` for any user-facing change.

## Git Workflow

- **Branch naming**: `CDS-<ticket>` or `CDS-<ticket>-short-description`
- **Commit format**: `feat|fix|techdebt|chore: [CDS-XXXXX]: Description`
- **PR title**: matches commit format
- **PR creation**: use `codepulse pr create`. Do not push branches and create PRs manually.
- **Default branch**: `main`

## DOs

- Always run `helm lint charts/` and `helm template charts/` before committing template changes.
- Bump `charts/Chart.yaml` `version:` on user-facing changes (templates, values defaults, sub-chart upgrades).
- Update `CHANGELOG.md` to document the bump.
- Reference the Jira ticket (`CDS-XXXXX`) in branch, commit, and PR title.
- When adding a new value to `values.yaml`, give it a sensible default and document it in the `README.md` if user-facing.
- Use `existingSecrets` patterns already established for any new secret-like input (see `README.md`).

## DON'Ts

- Never force-push to `main` (`git push --force origin main`).
- Never bypass hooks (`--no-verify`, `--no-gpg-sign`).
- Never commit secrets, tokens, or `.env` files.
- Never bump the `argo-cd` sub-chart version without rendering and testing the chart against a real cluster — breaking changes in argo-helm have shipped before.
- Never edit files under `charts/charts/` directly — that directory holds vendored sub-chart tarballs managed by `helm dependency update`.
- Never edit files under `releases/` or `index.yaml` by hand — the chart-releaser action owns them.

## Commands to Never Run

- `git push --force origin main`
- `git commit --no-verify` / `git push --no-verify`
- `rm -rf releases/` or `rm -rf charts/charts/`
- `helm package` directly into `releases/` (CI owns publishing)
